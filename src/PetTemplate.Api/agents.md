# Инструкции для работы с бекендом проекта PetTemplate

## Валидация

- Используем `ModelState` с `DataAnnotations` для валидации входных данных
- Контроллеры или отдельные эндпоинты помечаются атрибутом `[ValidateRequest]`
- **НЕ нужно** в теле методов контроллера дополнительно проверять `ModelState.IsValid`
- **НЕ указываем** тексты сообщений об ошибках в атрибутах валидации (например, `ErrorMessage`), так как они отображаются на фронтенде на языке интерфейса пользователя

## Аутентификация

- Используем атрибут `[AuthorizePermissions(PermissionEnum.)]` для всех эндпоинтов, которые являются специфичными для организаций
- В атрибут передаем необходимый список разрешений (permissions)
- **НЕ дублируем** проверку прав внутри сервисов - вся авторизация и проверка permissions должна происходить на уровне контроллера через атрибуты
- **ВАЖНО:** Для методов чтения (GET) необходимо указывать **оба** разрешения: `Crud` и `Read`. Например: `[AuthorizePermissions(PermissionEnum.organizationMemberCrud, PermissionEnum.organizationMemberRead)]`. Это связано с тем, что `Crud` подразумевает возможность просмотра и редактирования, поэтому пользователи с `Crud` должны иметь доступ к методам чтения

### Валидация контекста организации

- Атрибут `[AuthorizePermissions]` **автоматически** проверяет соответствие `OrganizationId` из параметров запроса с `OrganizationId` из хедера `X-Organization-Id`
- Проверка работает для:
  - Query параметров с именем `organizationId`
  - Свойства `OrganizationId` в DTO (для POST/PUT запросов)
- **НЕ нужно** писать вручную проверки `if(dto.OrganizationId != organizationIdFromHeader)` в каждом методе
- Если `OrganizationId` не совпадает, возвращается `403 Forbidden`

## Логирование и обработка ошибок

- У нас есть `GlobalExceptionHandler`, который автоматически обрабатывает все исключения
- **НЕ требуется** оборачивать код в контроллерах в дополнительные `try-catch` блоки для логирования
- Исключения из сервисов будут автоматически перехвачены и залогированы глобальным обработчиком

## Сервисы

- Если ожидается, что в базе будет **одна** сущность, используем `SingleAsync()`
- Если ожидается **ноль или одна** сущность, используем `SingleOrDefaultAsync()` с последующей проверкой на `null`
- **НЕ используем** `FirstOrDefaultAsync()` с последующим выбрасыванием `InvalidOperationException` - вместо этого применяем `SingleAsync()`, который сам выбросит исключение при отсутствии или множественности результатов

## Style Guide

- **НЕ используем** суффикс `Async` в названиях своих методов - все методы по умолчанию асинхронные
- В методах контроллеров всегда прокидываем параметр `CancellationToken ct`
- В методах сервисов также прокидываем `CancellationToken ct = default`
